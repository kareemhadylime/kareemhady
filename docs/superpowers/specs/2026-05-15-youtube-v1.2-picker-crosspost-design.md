# YouTube Integration — V1.2 (Picker / Cross-post) — Design Spec

**Date:** 2026-05-15
**Author:** kareemhady + Claude
**Status:** Draft, awaiting user review

## Goal

Build Phase V1.2 of the YouTube integration: enable operators to pick a published YouTube video from the @beithady channel and cross-post it as creative across **all five** existing ad publish paths in one click. The video can either live locally in Supabase (uploaded via V1.1's pipeline) or only exist on YouTube (uploaded via YouTube Studio directly).

| Phase | Scope | Status |
|---|---|---|
| V1.1 | Upload-out — file → YouTube with metadata | ✅ Shipped 2026-05-15 |
| **V1.2 (this spec)** | **Picker / cross-post for IG Reel + TikTok + Google PMax + Meta paid video ad** | Draft |
| V1.3 | Shortlinks — `app.limeinc.cc/yt/<slug>` redirects | Future |
| V1.4 | Gallery integration polish + cross-feature wrap-up | Future |

## Non-goals (V1.2)

- ❌ Add YT video to **existing** Google PMax campaign (V1.2 only creates new)
- ❌ Add YT video to **existing** Meta campaign / adset
- ❌ Auto-trim long-form videos to fit IG Reel format (operator handles in YouTube Studio editor)
- ❌ DB-cached YouTube channel sync (V1.2 uses 5min in-memory cache; V1.3 adds `ads_youtube_channel_cache` if picker latency matters)
- ❌ Multi-language metadata translation for cross-posts
- ❌ Cross-platform performance dashboard (V1.4 topic)
- ❌ Boost an organic IG Reel that was itself cross-posted from YouTube (operator chains V1.2 IG-Reel-cross-post → existing V1.1-era boost flow manually)

## Locked product decisions (from brainstorming)

| # | Decision | Rationale |
|---|---|---|
| Q1 | **Scope D — all three platforms** | Organic IG/TikTok + Google PMax (paid) + Meta paid video ad. Folded TikTok paid in via the 5 publish-page surface. |
| Q2 | **Picker location C — both standalone + embedded** | Standalone at `/beithady/gallery/youtube/picker` for browsing/discovery; embedded "Source: YouTube" tab on each of the 5 publish pages for in-flow swapping. |
| Q3 | **Picker source B — hybrid** | `ads_youtube_videos` rows MERGED with channel videos via `playlistItems.list` on the stored `youtube_uploads_playlist_id`. Action availability auto-degrades for YT-only rows (no Supabase bytes → only Google PMax works). |
| Q4 | **Format gating A — auto-hide incompatible actions** | Vertical Shorts get all 5 actions; horizontal long-form hides IG Reel + organic TikTok (keeps PMax + Meta paid + TikTok paid). |

## Architecture chosen: Approach 1 — Per-platform deep-link (lightweight)

Three approaches considered (per-platform deep-link / unified server action / monolithic cross-post page). Approach 1 picked because it:

- Mirrors V1.1's asset-modal-button → `?asset=<uuid>` pattern operators already know
- Each publish page does what it already does — just adds a small `?yt_video_id` pre-fill branch (~30 lines)
- New code is contained to one NEW lib (`meta-video-ad-publish.ts`) — the only platform without an existing fresh-creative pipeline

### Schema addition — migration `0135_ads_youtube_cross_posts.sql`

```sql
CREATE TABLE ads_youtube_cross_posts (
  id bigserial PRIMARY KEY,
  ads_youtube_video_id bigint REFERENCES ads_youtube_videos(id) ON DELETE CASCADE,
  -- nullable because YT-only rows (not in ads_youtube_videos) can still
  -- be cross-posted to Google PMax via youtube_video_id reference.
  youtube_video_id text NOT NULL,                -- always populated (canonical YT ID)
  target_platform text NOT NULL CHECK (target_platform IN (
    'instagram_reel', 'tiktok_organic', 'tiktok_paid', 'meta_video_ad', 'google_pmax'
  )),
  target_post_id bigint,                          -- loose FK to per-platform post tables (polymorphic)
  target_campaign_id bigint,                      -- loose FK for paid platforms
  status text NOT NULL DEFAULT 'queued'
    CHECK (status IN ('queued', 'published', 'error')),
  error text,
  created_by_user_id uuid REFERENCES auth.users(id),
  created_at timestamptz NOT NULL DEFAULT now(),
  UNIQUE (youtube_video_id, target_platform, target_post_id)
);

CREATE INDEX ads_youtube_cross_posts_video_idx
  ON ads_youtube_cross_posts (youtube_video_id, target_platform);

CREATE INDEX ads_youtube_cross_posts_ads_yt_video_idx
  ON ads_youtube_cross_posts (ads_youtube_video_id)
  WHERE ads_youtube_video_id IS NOT NULL;
```

The unique constraint enables O(log n) "already posted to X" lookups for the picker's badge UI.

## Picker module (`src/lib/beithady/youtube/picker.ts`)

### Public API

```ts
export type PickerItem = {
  // Identity
  youtube_video_id: string;          // YT canonical ID (e.g. '9fmAI8RJRr8')

  // Display
  title: string;
  description: string | null;
  thumbnail_url: string | null;      // YouTube's auto-generated thumbnail
  watch_url: string;                 // https://youtu.be/<id>
  duration_seconds: number | null;
  is_shorts: boolean;                // computed from duration ≤60 + (where known) vertical aspect
  privacy_status: 'public' | 'unlisted' | 'private' | 'unknown';

  // Building / origin
  building_code: string | null;      // from local DB only; null for YT-only rows

  // Stats (from V1.1 stats-sync cron)
  view_count: number;
  like_count: number;
  comment_count: number;
  stats_synced_at: string | null;

  // Source metadata
  in_local_db: boolean;
  ads_youtube_video_id: number | null;
  source_url: string | null;         // re-signed at picker load; null for YT-only rows
  published_at: string | null;

  // Action availability — computed in-module from format + source
  actions: Record<TargetPlatform, { available: boolean; reason?: string }>;

  // "Already posted" — from ads_youtube_cross_posts
  already_cross_posted: Record<TargetPlatform, { count: number; last_posted_at: string | null }>;
};

export type TargetPlatform =
  | 'instagram_reel' | 'tiktok_organic' | 'tiktok_paid' | 'meta_video_ad' | 'google_pmax';

export type PickerFilters = {
  building_code?: string | null;
  format?: 'shorts' | 'longform' | 'all';
  search?: string;
  sort?: 'recent' | 'views' | 'likes';
};

export async function listPickerVideos(
  accountId: number,
  filters?: PickerFilters
): Promise<PickerItem[]>;
```

### Action availability matrix

| Platform | Requires `source_url` (local bytes)? | Requires Shorts format? | Reason if blocked |
|---|---|---|---|
| `instagram_reel` | ✅ | ✅ (IG Reels = vertical ≤90s) | "Long-form" or "YouTube-only video — upload via app first" |
| `tiktok_organic` | ✅ | ✅ (TikTok organic optimized for vertical) | Same |
| `tiktok_paid` | ✅ | ❌ (TikTok ads accept landscape) | "YouTube-only video — upload via app first" |
| `meta_video_ad` | ✅ (Meta hosts video as ad creative — needs bytes) | ❌ | Same |
| `google_pmax` | ❌ (uses `youtube_video_id` reference) | ❌ | Always available |

### Cache strategy

- 5-minute in-memory cache keyed by `accountId` — wraps the YouTube API call (`playlistItems.list`) and the DB join, since both change slowly.
- Cache invalidation: TTL only (5 min). No explicit busting in V1.2.
- DB cache table deferred to V1.3 if picker latency becomes >500ms.

### YouTube API quota

`playlistItems.list` = 1 unit/call. 5min cache + 1 operator = ~288 calls/day = ~288 units. Well under the 10,000 daily quota.

## Per-platform cross-post flows

All five share the same entry pattern:
```
Picker → action button → /beithady/ads/<platform>?yt_video_id=42&source=youtube
  → publish page reads param, re-signs source_url, pre-fills form
  → operator confirms → Publish
  → existing publish lib runs (or new meta-video-ad-publish.ts)
  → success: recordCrossPost(...) writes audit row
```

### 3a — Organic IG Reel

**Page:** `/beithady/ads/instagram/reels?yt_video_id=42`
**Lib:** `publishInstagramReel({ videoUrl, caption, ... })` — already accepts HTTPS URL, no lib change.
**Pre-fill:** `videoUrl = ads_youtube_videos.source_url` (re-signed), `caption = YT title + first 200 chars description`, `hashtags = trailing hashtag block from description`.
**Audit row:** `target_platform='instagram_reel'`, `target_post_id=ads_instagram_posts.id`.

### 3b — Organic TikTok

**Page:** `/beithady/ads/tiktok/organic?yt_video_id=42`
**Lib:** `publishTikTokOrganic` (V1.1 FILE_UPLOAD pipeline).
**Pre-fill:** Same shape as 3a — uses `source_url`.
**Caveat:** TikTok Content Posting API still requires Direct Post audit (V1.1 status — submitted; posts go to TikTok Inbox until audit clears, operator finalizes in TikTok app).
**Audit row:** `target_platform='tiktok_organic'`, `target_post_id=ads_tiktok_posts.id`.

### 3c — TikTok paid

**Page:** `/beithady/ads/tiktok/paid?yt_video_id=42`
**Lib:** Existing `tiktok-paid-publish.ts`.
**Pre-fill:** Video asset field gets `source_url`. Operator fills budget / targeting / objective.
**Audit row:** `target_platform='tiktok_paid'`, `target_campaign_id=...`.

### 3d — Meta paid video ad (NEW pipeline)

**Page:** `/beithady/ads/instagram/boost?yt_video_id=42`. Existing flow boosts an existing IG organic post; YouTube source forks into a new "fresh video ad" path.

**New lib:** `src/lib/beithady/ads/meta-video-ad-publish.ts` (~250 LOC).

```
1. POST /act_{ad_account_id}/advideos (multipart, our source_url bytes)
   → returns { id: video_id }
2. Poll /{video_id}?fields=status until status='ready' (≤30s typically)
3. POST /act_{ad_account_id}/campaigns (objective, status=PAUSED)
4. POST /act_{ad_account_id}/adsets (targeting, daily_budget, optimization_goal)
5. POST /act_{ad_account_id}/adcreatives (video_data: { video_id, image_url=YT thumbnail, call_to_action })
6. POST /act_{ad_account_id}/ads (creative_id, adset_id)
All entities created PAUSED — operator activates in Ads Manager.
```

**Why new lib instead of extending boost-publish.ts:** boost takes `object_story_id` (existing IG media). Fresh ad uses `advideos` + `adcreative.video_data` — different Meta API surface entirely.

**Page UX:** boost page gets a tab/branch — "Source: existing IG post" (existing) vs "Source: YouTube video" (new). YouTube source swaps form to new pipeline inputs.

**Audit row:** `target_platform='meta_video_ad'`, `target_campaign_id`.

### 3e — Google PMax with YouTube video asset

**Page:** `/beithady/ads/google/pmax?yt_video_id=42`
**Lib:** Extend `google-pmax-publish.ts` (~50 LOC). After existing assetGroupAssets:mutate step, add:

```ts
if (input.youtube_video_id) {
  // 1. Create YouTube video asset
  const assetResp = await gadsMutate(customerId, [{
    create: {
      asset: {
        youtube_video_asset: { youtube_video_id: input.youtube_video_id },
        type: 'YOUTUBE_VIDEO',
      },
    },
  }]);
  const assetResourceName = assetResp.results[0].resource_name;

  // 2. Link to asset group
  await gadsMutate(customerId, [{
    create: {
      asset_group_asset: {
        asset_group: assetGroupResourceName,
        asset: assetResourceName,
        field_type: 'YOUTUBE_VIDEO',
      },
    },
  }]);
}
```

**Crucial difference from other platforms:** no bytes transferred. Video stays on YouTube. Works for **YT-only rows** (the only platform that does).

**Audit row:** `target_platform='google_pmax'`, `target_campaign_id`.

### Shared `recordCrossPost` helper

```ts
// src/lib/beithady/youtube/cross-post-audit.ts
export async function recordCrossPost(input: {
  ads_youtube_video_id: number | null;
  youtube_video_id: string;
  target_platform: TargetPlatform;
  target_post_id?: number | null;
  target_campaign_id?: number | null;
  status: 'published' | 'error';
  error?: string | null;
  created_by_user_id?: string | null;
}): Promise<void>;
```

Best-effort insert — wrapped in try/catch and logged but not rethrown. Publish success isn't blocked by audit row failure.

## Error handling

### Picker-side errors

| Error | Surface | Action |
|---|---|---|
| YouTube API `quotaExceeded` | Toast: "YouTube channel sync unavailable — showing local DB videos only" | Wait until UTC midnight (quota resets). Picker degrades gracefully. |
| YouTube auth `invalid_grant` | Banner above picker: "@beithady token expired — reconnect" + link to `/beithady/ads/accounts` | Click Reconnect (V1.1 hardened pattern). |
| Supabase query failure | Inline: "Failed to load picker — refresh to retry" | Refresh |
| Empty result set | Empty state: "No YouTube videos yet — [Upload your first one →]" | Upload via V1.1 publish page |

### Pre-publish errors

| Error | Surface | Action |
|---|---|---|
| `yt_video_id` not found | Publish banner: "YouTube video #42 not found" + Back to picker | Back to picker |
| `source_url` 404 / unreachable | Error card: "Source video unavailable — Supabase storage may have been pruned" | Re-upload via app |

### Per-platform publish errors

Each platform's existing error path triggers. Cross-post audit row gets `status='error'` and `error=<platform message>`.

### Audit row failure

Best-effort. Wrapped in try/catch. Doesn't block publish success.

```ts
try {
  await recordCrossPost({ ... });
} catch (e) {
  console.error('[yt-cross-post] audit insert failed', e);
}
```

Unique constraint duplicate fires fail-soft.

### Typed error classes (new)

- `PickerSourceUnavailableError(yt_video_id, reason: 'not_found' | 'source_url_expired' | 'yt_only_no_bytes')`
- `CrossPostAuditError(message)` — non-fatal
- `MetaVideoUploadError(step: 'advideos' | 'status_poll' | 'campaign' | 'adset' | 'creative' | 'ad', message)`
