# YouTube Integration — V1.1 (Upload-out) — Design Spec

**Date:** 2026-05-15
**Author:** kareemhady + Claude
**Status:** Draft, awaiting user review

## Goal

Connect the Beithady module of the Lime Investments dashboard to a YouTube channel (initially `@beithady`) so operators can upload videos from the app with title / description / tags / language / privacy metadata, without ever leaving the dashboard. This is **Phase V1.1** of a four-phase YouTube rollout:

| Phase | Scope |
|---|---|
| **V1.1 (this spec)** | Upload-out — push a video file from the app to YouTube with metadata |
| V1.2 | Picker / cross-post — use existing YouTube videos as a source for Meta / TikTok / Google ad pipelines |
| V1.3 | Shortlinks — `app.limeinc.cc/yt/<slug>` redirects to YouTube watch URLs, with click tracking |
| V1.4 | Gallery integration polish + cross-feature wrap-up |

Each subsequent phase will get its own spec → plan → implementation cycle. This spec only defines V1.1.

## Non-goals (V1.1)

- ❌ Playlist auto-add (V1.4 — needs broader `youtube` OAuth scope)
- ❌ Custom thumbnails (V1.5 — `thumbnails.set` + image crop UI)
- ❌ Scheduled publishing (`publishAt`) (V1.5)

- ❌ Bulk upload (V1.5)

## Locked product decisions (from brainstorming)

| # | Decision | Rationale |
|---|---|---|
| Q1 | **Gallery-first** for video source | Disk uploads pass through Supabase Gallery before YouTube. Keeps Gallery as the canonical master library so V1.2's picker naturally sees every YT video as a Gallery asset. No YT-only orphans. |
| Q2 | **Single Beit Hady channel** for V1.1 | But schema uses multi-row `ads_accounts` so V1.5 can grow to Boat Rental + Kika without a migration. |
| Q3 | **Both long-form and Shorts** supported | Form toggles validation; vertical ≤60s gets `#Shorts` auto-injected into description's first line. |
| Q4 | **AI + template hybrid metadata** | Operator picks a building/format template; Claude vision fills `{variables}` from a sampled frame; operator reviews/edits before submit. |
| Q5 | **Default privacy: `unlisted`** | Operator-overridable. Unlisted videos are link-accessible (V1.3 shortlinks work immediately) but hidden from channel page until the operator promotes to `public`. |
| Q6 | **Three UI entry points** | Standalone publish page at `/beithady/gallery/youtube/`, asset-modal button on each video asset, and Gallery landing tile. |

**Stats tracking** (added by kareem on review of § 2): persist `view_count` / `like_count` / `comment_count` on every published row; refresh every 6h via cron.

**Language tracking** (added on review of § 2): `language text DEFAULT 'en'` (BCP-47 like `en`, `ar`, `en-US`). Operator sets per video; AI infers from visual cues.

## Architecture chosen: Approach 3 — Hybrid sync/async

After considering three architectures (pure sync server action; pure async cron queue; hybrid), V1.1 uses the **hybrid** approach:

- **Sync path** for Shorts ≤ 60s and file size ≤ 200 MB — operator gets the watch URL within 30–90 s.
- **Async cron-driven queue** for long-form or files > 200 MB — operator submits and tracks status on the publish page; cron does the upload work.

This matches Beit Hady's expected mix (mostly Shorts, occasional property-tour long-form) and avoids Vercel's serverless function timeout on long uploads without paying the latency cost on Shorts.

### State machine

```
        ┌─ sync path: rows transition in-place ─┐
        │                                        │
queued ──┴──> uploading ──> processing ──> published ──┐
                                              ▲        │
                                              │        │  6h
queued ──> uploading ──> [chunk loop] ────────┘    [stats-sync cron]
   │           │                                       refreshes
   │           │                                       view_count,
   └──[cron retries with backoff]──> error             like_count,
                                  (5 retries terminal) comment_count
```

### Design judgement calls

1. **Partial index on `(status, next_retry_at)`** — keeps the cron query O(in-flight rows), not O(table). Mirrors the Meta cron pattern.
2. **`asset_id` is `ON DELETE SET NULL`** — keep video rows for audit even if Gallery asset gets nuked later.
3. **No FK on `building_code`** — kept as text tag matching existing pattern in `ads_instagram_posts` and `ads_tiktok_posts`.
4. **`source_url` is a signed URL, not a storage key** — YouTube's resumable session URL expires in ~7 days. We persist the signed URL (refresh by re-signing if expired before cron picks the row up).
5. **`upload_session_url` lifetime: 7 days** — if `now() - created_at > 6 days`, restart the session fresh rather than retry.

### Flow walkthrough

```
1. Operator clicks "Connect →" on /beithady/ads/accounts (YouTube row)
   ↓
2. GET /api/auth/google-youtube/start?account_id=<id>
   • Generates 16-byte CSRF token
   • Sets cookie  oauth_yt_state=<csrf>  (httpOnly, secure, 10min TTL)
   • Redirects to Google with:
       state = `${csrf}.${account_id}`
       scope = youtube.upload + youtube.readonly
       access_type = offline           ← forces refresh token
       prompt = consent                ← FORCES refresh token return on re-auth
       include_granted_scopes = true
   ↓
3. Operator consents on Google's screen ("Unverified app" warning on first run — fine for single-operator)
   ↓
4. Google redirects to /api/auth/google-youtube/callback?code=…&state=…
   • Verifies cookie csrf matches state prefix → else 400
   • Extracts account_id from state suffix
   • POSTs code → oauth2.googleapis.com/token
       → { access_token, refresh_token, expires_in }
   • Calls youtube.googleapis.com/youtube/v3/channels?part=snippet,contentDetails&mine=true
       → captures channel.id, channel.snippet.title, snippet.customUrl,
                   contentDetails.relatedPlaylists.uploads
   • Encrypts both tokens via src/lib/crypto.ts (AES-256-GCM)
   • UPDATE ads_accounts SET
       youtube_channel_id              = <UCxxx>,
       youtube_channel_name            = <title>,
       youtube_channel_handle          = <customUrl>,        -- @beithady
       youtube_refresh_token           = <encrypted>,
       youtube_access_token            = <encrypted>,
       youtube_access_token_expires_at = now() + expires_in,
       youtube_uploads_playlist_id     = <UUxxx>
     WHERE id = <account_id>
   • Redirects to /beithady/ads/accounts?connected=youtube
```

### Token refresh + reconnect UX

Mirrors the **`tiktok-client.ts` pattern** (which was hardened during the May 14 `refresh_failed` debugging):

```ts
async function getYouTubeAccessToken(accountId: number): Promise<string> {
  const row = await loadAccount(accountId);

  // 1. Cached access token still valid? Skip refresh.
  if (row.youtube_access_token_expires_at &&
      new Date(row.youtube_access_token_expires_at) > new Date(Date.now() + 60_000)) {
    return decrypt(row.youtube_access_token!);
  }

  // 2. Refresh.
  const storedRefresh = unwrapStoredRefreshToken(row.youtube_refresh_token);  // try-decrypt fallback
  const res = await fetch('https://oauth2.googleapis.com/token', {
    method: 'POST',
    body: new URLSearchParams({
      client_id: process.env.GOOGLE_CLIENT_ID!,
      client_secret: process.env.GOOGLE_CLIENT_SECRET!,
      refresh_token: storedRefresh,
      grant_type: 'refresh_token',
    }),
  });
  const json = await res.json();

  // 3. Refresh failed? Clear dead token + surface refresh_failed.
  if (json.error === 'invalid_grant') {
    await clearDeadToken(accountId);
    throw new YouTubeAuthError('refresh_failed');
  }

  // 4. Store rotated tokens (Google may rotate; always re-encrypt).
  await updateTokens(accountId, json.access_token, json.refresh_token, json.expires_in);
  return json.access_token;
}
```

**Reconnect UX**: `/beithady/ads/accounts` shows a yellow "Reconnect" link beside "Configure" when `youtube_refresh_token` is null OR when the publish form surfaces a `refresh_failed` error code.

## Upload pipeline

### Sync path (Shorts ≤ 60 s & ≤ 200 MB)

Lives in `publishYouTubeVideoAction` server action at `src/app/beithady/gallery/youtube/actions.ts`. Route declares `export const maxDuration = 300`.

```ts
async function publishSync(input): Promise<PublishedVideo> {
  const accessToken = await getYouTubeAccessToken(input.account_id);
  const bytes = await fetchVideoBytes(input.source_url);   // returns ArrayBuffer

  // STEP 1 — POST initiates the resumable session with metadata
  const initResp = await fetch(
    'https://www.googleapis.com/upload/youtube/v3/videos?uploadType=resumable&part=snippet,status',
    {
      method: 'POST',
      headers: {
        'Authorization': `Bearer ${accessToken}`,
        'Content-Type': 'application/json; charset=UTF-8',
        'X-Upload-Content-Length': String(bytes.byteLength),
        'X-Upload-Content-Type': 'video/mp4',
      },
      body: JSON.stringify({
        snippet: {
          title: input.title,
          description: input.description,
          tags: input.tags,
          categoryId: String(input.category_id),
          defaultLanguage: input.language,
          defaultAudioLanguage: input.language,
        },
        status: {
          privacyStatus: input.privacy_status,
          selfDeclaredMadeForKids: false,
          embeddable: true,
        },
      }),
    }
  );
  const sessionUrl = initResp.headers.get('Location');

  // STEP 2 — PUT all bytes in one shot
  const uploadResp = await fetch(sessionUrl, {
    method: 'PUT',
    headers: {
      'Content-Length': String(bytes.byteLength),
      'Content-Range': `bytes 0-${bytes.byteLength - 1}/${bytes.byteLength}`,
    },
    body: bytes,
  });
  const video = await uploadResp.json();

  return {
    video_id: video.id,
    watch_url: `https://youtu.be/${video.id}`,
    upload_status: video.status?.uploadStatus,
  };
}
```

**Operator UX**: spinner for ~30–90 s, then redirect to publish page with `?published=<id>` showing the watch URL.

### Async path (long-form OR > 200 MB)

`publishYouTubeVideoAction` just inserts the `ads_youtube_videos` row with `status='queued'`. Cron does the work.

**Cron handler** `src/app/api/cron/youtube-uploader/route.ts`, registered in `vercel.json` every minute:

```ts
export const maxDuration = 800;

export async function GET(req: Request) {
  if (!isCronAuthed(req)) return new Response('unauthorized', { status: 401 });

  // Pick up to 3 in-flight rows. With 800s budget shared across rows.
  const rows = await sb.from('ads_youtube_videos')
    .select('*')
    .in('status', ['queued', 'uploading', 'processing'])
    .or('next_retry_at.is.null,next_retry_at.lte.now()')
    .order('id')
    .limit(3);

  for (const row of rows) {
    try {
      if (row.status === 'queued')        await initSession(row);
      else if (row.status === 'uploading')  await sendChunksUntilBudget(row);
      else if (row.status === 'processing') await pollProcessing(row);
    } catch (e) {
      await scheduleRetry(row, e);  // backoff 2,4,8,16,32 min; status='error' after 5
    }
  }
}
```

#### Chunk strategy

- **Chunk size: 8 MiB** (multiple of 256 KB as required by YouTube; large enough for throughput, small enough that each PUT is sub-10s).
- **Bytes source**: HTTP `Range: bytes=offset-end` request against the Supabase signed URL. Supabase Storage supports range responses. We never materialize the full file in cron memory.
- **Per-cron-invocation budget**: send chunks for up to ~700 s, then save `chunk_offset` and exit. Next cron picks up. For a 1 GB file at 50 Mbps that's ~160 s — finishes in one invocation. A 5 GB file might need two.

```ts
async function sendChunksUntilBudget(row) {
  const startMs = Date.now();
  const BUDGET_MS = 700_000;
  const CHUNK = 8 * 1024 * 1024;
  let offset = row.chunk_offset;
  const total = row.file_size_bytes;

  while (offset < total && (Date.now() - startMs) < BUDGET_MS) {
    const end = Math.min(offset + CHUNK, total);
    const chunkBytes = await fetchRangedBytes(row.source_url, offset, end - 1);

    const resp = await fetch(row.upload_session_url, {
      method: 'PUT',
      headers: {
        'Content-Length': String(end - offset),
        'Content-Range': `bytes ${offset}-${end - 1}/${total}`,
      },
      body: chunkBytes,
    });

    if (resp.status === 308) {
      offset = parseRangeEnd(resp.headers.get('Range')) + 1;
      await sb.from('ads_youtube_videos').update({ chunk_offset: offset }).eq('id', row.id);
    } else if (resp.status === 200 || resp.status === 201) {
      const video = await resp.json();
      await sb.from('ads_youtube_videos').update({
        status: 'processing',
        youtube_video_id: video.id,
        watch_url: `https://youtu.be/${video.id}`,
        chunk_offset: total,
      }).eq('id', row.id);
      return;
    } else {
      throw new YouTubeUploadError(`chunk_failed: ${resp.status} ${await resp.text()}`);
    }
  }
}
```

### `processing → published` polling

Once bytes are uploaded, YouTube is transcoding/generating thumbnails. Cron polls `videos.list?id=…&part=status` once per iteration:

```ts
async function pollProcessing(row) {
  const accessToken = await getYouTubeAccessToken(row.account_id);
  const resp = await fetch(
    `https://www.googleapis.com/youtube/v3/videos?id=${row.youtube_video_id}&part=status,snippet`,
    { headers: { 'Authorization': `Bearer ${accessToken}` } }
  );
  const { items: [video] } = await resp.json();

  if (video.status.uploadStatus === 'processed') {
    await sb.from('ads_youtube_videos').update({
      status: 'published',
      published_at: new Date().toISOString(),
      thumbnail_url: video.snippet.thumbnails?.high?.url,
    }).eq('id', row.id);
  } else if (video.status.uploadStatus === 'failed') {
    await sb.from('ads_youtube_videos').update({
      status: 'error',
      error: video.status.failureReason || 'youtube_processing_failed',
    }).eq('id', row.id);
  }
  // else: still processing, leave; cron picks up next minute
}
```

### Retry + backoff

- **Transient errors** (5xx, 429, network): bump `retry_count`, set `next_retry_at = now() + 2^retry_count * 1 min` (so 2, 4, 8, 16, 32 min).
- **Auth errors (401 `invalid_token`)**: refresh access token in-flight, retry once.
- **Refresh errors (`invalid_grant`)**: clear `youtube_refresh_token`, set `status='error'`, surface "refresh_failed" → operator clicks Reconnect.
- **Quota errors (`quotaExceeded`, 403)**: set `next_retry_at = tomorrow 00:00 UTC` (quota resets daily).
- **Terminal**: after 5 retries → `status='error'`, operator sees the error message in the publish page table with a "Retry" button (resets `retry_count=0, next_retry_at=now()`).

### Sync vs async branching

```ts
const SYNC_DURATION_MAX = 60;
const SYNC_SIZE_MAX = 200 * 1024 * 1024;

const useSync = (input.duration_seconds ?? 9999) <= SYNC_DURATION_MAX
             && input.file_size_bytes <= SYNC_SIZE_MAX;
```

### Row lifecycle (both paths use the same row, same table)

Both paths insert the `ads_youtube_videos` row at submit time so the Recent uploads table reflects the upload from the moment the operator clicks Publish. The only difference is which status it starts in:

**Sync path:**
1. Server action inserts row with `status='uploading'`, all operator inputs, `asset_id` (if from Gallery).
2. Call YouTube API (init + PUT all bytes). Sync server action blocks here.
3. On success: UPDATE row to `status='processing'`, set `youtube_video_id`, `watch_url`, `chunk_offset = file_size_bytes`.
4. On error: UPDATE row to `status='error'`, set `error`. Sync action returns error to form.
5. Stats-sync cron picks up `processing` → `published` transition (same code path as async).

**Async path:**
1. Server action inserts row with `status='queued'`, all operator inputs.
2. Returns immediate redirect to Recent uploads with `?queued=<row.id>`.
3. Cron drives state machine: `queued` → `uploading` → `processing` → `published` (or `error` after 5 retries).

### Stats-sync cron (every 6 h)

`src/app/api/cron/youtube-stats-sync/route.ts`, schedule `0 */6 * * *`:

- Picks `status='published'` rows where `stats_synced_at IS NULL OR stats_synced_at < now() - interval '6 hours'`.
- Batches in groups of 50 (`videos.list?part=statistics&id=` accepts comma-separated, 1 quota unit per call regardless of batch size).
- Updates `view_count`, `like_count`, `comment_count`, `stats_synced_at`.

**Quota math**: YouTube Data API daily quota = 10,000 units. At 1,000 published videos × 4 refreshes/day × (1000/50) = 80 units/day. Well under cap.

### Frame sampling — client-side, single midpoint frame

```ts
async function captureMidFrame(videoFile: File): Promise<string /* dataURL */> {
  const video = document.createElement('video');
  video.src = URL.createObjectURL(videoFile);
  await new Promise(r => video.onloadedmetadata = r);
  video.currentTime = video.duration / 2;
  await new Promise(r => video.onseeked = r);

  const canvas = document.createElement('canvas');
  canvas.width = video.videoWidth;
  canvas.height = video.videoHeight;
  canvas.getContext('2d')!.drawImage(video, 0, 0);
  URL.revokeObjectURL(video.src);
  return canvas.toDataURL('image/jpeg', 0.7);   // ~30-80KB at 1080p
}
```

**Why client-side**: operator already has the file loaded in the browser for upload progress. Re-decoding server-side would be a redundant download + transcode. A single canvas snap is ~50KB JPEG, no extra deps.

**Why single frame** (not multi): a midpoint frame catches the main subject for ~95% of property tour content. Multi-frame sampling deferred to V1.5 if AI quality is poor in practice.

## Error handling

| Category | Where it surfaces | Operator action |
|---|---|---|
| Form validation (missing title, file too big) | Inline field error + disabled submit | Fix the form |
| OAuth `invalid_grant` | Red banner: "@beithady token expired" + Reconnect button | Click Reconnect → re-OAuth |
| AI generation failed (timeout, invalid JSON) | Toast "AI assist unavailable; using template defaults" — form continues | Fill manually |
| Sync upload network error | Inline error: "Upload failed — N retries remaining" + Retry button | Click Retry (or check connection) |
| Async cron error (transient, retrying) | Row shows "⏳ retry in 4m" with `error` tooltip on hover | Wait |
| Async cron error (terminal, 5 retries) | Row shows "❌ error" + Retry button | Click Retry (resets `retry_count`) |
| YouTube content rejected (`status.rejectionReason`) | Row shows specific rejection (copyright, etc.) | Edit content or appeal on YouTube |
| Quota exceeded | Banner: "Daily YouTube quota hit; retrying tomorrow" | Wait |
| Refresh failed mid-upload | Same as `invalid_grant`; row stays in `uploading` until reconnect | Reconnect; cron retries automatically |
